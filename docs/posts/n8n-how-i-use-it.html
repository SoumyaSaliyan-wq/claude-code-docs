How I use n8n to stop
doing the same thing twice.

n8n is the automation tool I kept coming back to after trying everything else. Here's how I actually use it — not a feature tour, just the workflows that stuck.

7 min read · Practical

01 · Why

Why n8n and not something else

I'd tried building automations with custom scripts before. They work until you're the only one who can maintain them, and they break quietly. I wanted something visual enough to reason about at a glance, but that didn't take away the ability to write real logic when I needed it.

n8n hit that balance. It's visual, but it lets me write JavaScript or Python inside any node when the GUI isn't enough. I self-host it, which means my data stays where I want it and I'm not paying per task.

Before n8n

A mix of cron scripts and a growing list of "I'll automate that eventually" tasks that never got done.

After n8n

One place to see, debug, and reuse every automation I've built — and a much shorter list of things I still do manually.

02 · Setup

Running n8n locally with Docker

The quickest way to get n8n running is a single Docker command. No account needed, no cloud, everything local.

First, create a named Docker volume so your workflows and credentials survive container restarts:

Step 1 — create the volume

docker volume create n8n_data

Then start the container:

Step 2 — run n8n

docker run -it --rm \
  --name n8n \
  -p 5678:5678 \
  -e GENERIC_TIMEZONE="Asia/Kolkata" \
  -e TZ="Asia/Kolkata" \
  -e N8N_ENFORCE_SETTINGS_FILE_PERMISSIONS=true \
  -e N8N_RUNNERS_ENABLED=true \
  -v n8n_data:/home/node/.n8n \
  docker.n8n.io/n8nio/n8n

Open http://localhost:5678 and you're in. A few things worth noting about these flags:

↳

GENERIC_TIMEZONE + TZ

Sets the timezone for both n8n's scheduler and the underlying system. Without this, cron triggers run on UTC and your "every morning at 9am" workflow fires at the wrong time.

↳

N8N_RUNNERS_ENABLED

Enables the task runner mode introduced in recent versions — runs Code node executions in a separate process, which is more stable and avoids blocking the main n8n process.

↳

n8n_data named volume

Using a named volume instead of a bind mount means Docker manages the storage. Your workflows, credentials, and settings persist even after the container is removed and recreated.

Keep credentials safe. Everything n8n stores — credentials, workflow history, settings — lives inside that named volume. Back it up with docker run --rm -v n8n_data:/data alpine tar czf - /data before upgrading or wiping containers.

03 · What I build

The workflows I actually run

These aren't showcase automations — they're the ones still running months later because they genuinely save me time.

📊

Scheduled reports

Runs on a cron schedule, queries an internal database, shapes the results with a Code node, and delivers a formatted report — to Slack, email, or a shared doc — without anyone having to manually pull data.

🪣

S3 data pipelines

Picks up files from AWS S3 on a schedule or trigger, processes them — parse, transform, validate — and pushes results downstream to a database or another service. No custom Lambda needed.

🔌

Third-party API ingestion

Polls external APIs on a schedule, handles pagination and auth refresh automatically, and loads the data into an internal store. The HTTP Request node covers anything without a native n8n integration.

🔔

Alerts & thresholds

Checks a metric or condition on a schedule — database query, API response, file count — and fires a Slack alert or email the moment something goes outside the expected range.

🔍

Scheduled scans

Runs a sweep across a data source or set of endpoints, flags anomalies or missing records, and surfaces a summary. Useful for data quality checks that would otherwise be a manual spot-check.

🧩

Sub-workflow library

Common logic — auth refresh, retry with backoff, Slack alert — lives in reusable sub-workflows. Any workflow can call them like a function so I'm not copy-pasting the same nodes everywhere.

04 · Local dev

Building workflows with Claude Code + n8n

Once n8n is running locally, you can pair it with Claude Code to build workflows faster than clicking through the UI. The setup is straightforward: Claude Code connects to your local n8n instance via the n8n MCP server, and with a local API key it can create, update, and trigger workflows directly from the conversation.

What you need: n8n running on localhost:5678, an n8n API key (Settings → API → Create), and the n8n MCP configured in Claude Code's settings.

The n8n MCP gives Claude Code a set of tools — list workflows, get a workflow's JSON, create or update a workflow, execute a run. Combined with the n8n-specific skills available in Claude Code (expression syntax, node configuration, workflow patterns), you can describe what you want in plain language and have a working workflow scaffold ready to review.

Without this setup

Open the n8n canvas, drag nodes one by one, look up expression syntax, manually wire connections, test, fix, repeat.

With Claude Code + n8n MCP

"Create a workflow that polls this API every hour and posts to Slack if anything changed." — it writes the JSON, creates it in n8n, and you just review.

The n8n skills are what make this accurate rather than just fast. Claude Code has built-in knowledge of n8n's expression syntax ($json, $input, $now), common node configurations, and proven workflow patterns — so it doesn't just generate a plausible-looking workflow, it generates one that actually runs.

05 · AI workflows

Where the AI Agent node changed things

The part that surprised me most about n8n isn't the integrations — it's the AI Agent node. You wire up an LLM, attach tools (HTTP requests, database lookups, sub-workflows), and the agent decides which tools to call and in what order. You don't write the orchestration logic, you just define what's available.

Example: Research agent flow

The practical win: I stopped writing orchestration code. The agent handles the loop of "search → read → decide if I need more → summarise". I just define the tools, set a system prompt, and let it run. For research-style tasks this alone replaced a non-trivial amount of custom code.

06 · Tips

Things that took me too long to learn

01

Build one workflow for one job

The temptation is to make one mega-workflow that handles everything. Don't. Small, single-purpose workflows are easier to debug, easier to reuse, and don't break each other when you make changes. Use sub-workflows to share logic.

02

Use the Error Workflow feature early

Set up a dedicated error workflow that sends a Slack message whenever anything fails. Without it, workflows fail silently and you find out a week later that nothing has been running. Takes 5 minutes to set up, saves hours of confusion.

03

The Code node is not a last resort

Reach for a Code node early when the built-in nodes would need three steps to do what items.map() can do in one. It keeps workflows shorter and the logic is much easier to read.

04

Pin data during development

In the editor you can pin the output of any node so it doesn't re-fetch live data every time you run. Use this constantly when building — it speeds up iteration and stops you from hitting API rate limits while testing.

05

Webhook → respond immediately, process async

If a webhook caller is waiting for a response, return 200 OK right away with a Respond to Webhook node, then continue processing. Don't make the caller wait for your entire workflow to finish.
