AI · Tool Calling

Not just answers.
Actions.

LLMs are great at text. But the real world needs actions — fetching data, running code, calling APIs. Tool calling is how that gap gets closed. Here's how it works and when to reach for it.

6 min read · Beginner–Intermediate

01 · The Problem

What an LLM can't do on its own

A language model is, at its core, a next-token predictor. It's very good at reasoning, writing, and synthesising — but it has hard limits. It doesn't know what time it is. It can't fetch a URL. It can't run your code. It can't send an email.

Its knowledge is frozen at a training cutoff. Ask it "what's the weather in Austin right now?" and the best it can do is tell you about Austin's climate in general — it has no access to live data.

LLM alone

"I don't have access to real-time data, but Austin in July is typically hot and humid, with temperatures around 95–100°F."

LLM + tool calling

"Right now in Austin it's 98°F with a 40% chance of rain this afternoon. You'll want an umbrella."

Tool calling is the mechanism that bridges this gap — it lets the model reach out and do something in the real world, then use the result to answer you properly.

02 · What

What tool calling actually is

Tool calling (also called "function calling") is a way to give a language model a menu of functions it can invoke. You describe each tool — its name, what it does, and what parameters it takes — and the model decides when to call one and with what arguments.

Key insight: The model doesn't execute the tool itself. It outputs a structured request saying "call this function with these arguments." Your code runs the function and feeds the result back. The model then uses that result to produce its final answer.

Think of it like this: the model is a very smart analyst who can't use a computer. You're the assistant sitting next to them. When they need data, they tell you what to look up — you look it up and hand them the result. They do the thinking; you do the fetching.

03 · How

The tool calling loop

There's a specific back-and-forth that happens under the hood every time a tool is called. Understanding this loop makes everything else click.

A tool definition has three parts the model reads: a name, a description (this is the most important — the model uses it to decide when to call the tool), and a parameters schema that specifies what arguments to pass.

{
  "name": "get_weather",
  "description": "Get the current weather for a given city. Use this when the user asks about current weather conditions or forecasts.",
  "parameters": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "The city name, e.g. Austin"
      }
    },
    "required": ["city"]
  }
}

Notice the description isn't just "get weather" — it explicitly tells the model when to use this tool. That specificity matters a lot. The model makes its tool selection decision based on how well the description matches the user's intent.

04 · Use Cases

What you can actually do with it

Tool calling unlocks four broad categories of capability that a plain LLM can't handle.

🌐 Live data

Fetch real-time information the model's training doesn't include — weather, stock prices, news, live database queries.

🧮 Deterministic computation

Run calculations, parse dates, convert currencies. Don't trust a model to do math — trust a calculator.

✍️ Side effects

Write to a database, send an email, create a Jira ticket, push a file to S3. Actions that change state in the world.

🔗 Multi-step workflows

Chain multiple tool calls together — search for a user, look up their orders, then send them a refund. The model orchestrates the steps.

05 · When

When to use it — and when not to

Tool calling adds latency and complexity. It's not always the right answer.

✓

Use it when the answer depends on real-time or user-specific data

If the answer changes depending on when or who asks — current stock price, a user's account balance, today's schedule — you need a tool. The model can't reason about data it's never seen.

✓

Use it when you need a side effect

Sending a message, saving a record, triggering a workflow — anything that needs to actually happen in a system requires a tool. The model alone can only describe what to do, not do it.

✓

Use it when you need guaranteed precision

LLMs can hallucinate math, dates, and unit conversions. If precision is critical, use a tool that runs actual code — calculate(), parse_date() — and pass the result to the model.

✗

Skip it for static knowledge questions

"What's the capital of France?" doesn't need a tool — the model already knows. Adding unnecessary tools creates decision overhead and extra latency for no benefit.

✗

Skip it when the description is too vague to be reliable

If you can't write a clear one-sentence description of exactly when a tool should be called, the model won't use it reliably either. Fix the tool design first.

06 · What's next

Tool calling is the foundation of AI agents

Once you understand tool calling, everything in the AI agent space starts to make more sense. An agent is essentially a model in a loop — it calls tools, observes results, and decides what to do next until the task is done.

This is also where MCP (Model Context Protocol) fits in. MCP is a standardisation layer on top of tool calling — a shared protocol so tools can be defined once and used across different models and clients without rewiring everything. Think of tool calling as the concept, and MCP as the ecosystem built around it.

The pattern scales: A single tool call answers a question. A chain of tool calls completes a task. An agent with tools and a loop handles a workflow. Same primitive, different scale.

The description is the most important thing you write. Not the code that runs the tool — the two sentences that tell the model when to reach for it. That's the lever everything else turns on.
