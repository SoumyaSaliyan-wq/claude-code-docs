Claude Code · Real workflow

The markdown files
did the heavy lifting.

I had a hypothesis about a third-party API: different filters probably return different data. The documentation was a URL. In one session, Claude turned a PDF export of those docs into organised markdown files — and then into a Python script that told me exactly which filter gives you what.

5 min read · Backend developers, QA engineers

01 · Context

Not a bug hunt. A behaviour check.

We were working with a third-party API that supported a range of query filters. On paper they all looked like they'd give you the same data — just different ways to slice it. But I had a hunch that wasn't true. Different filters probably return different dataset shapes, different record counts, different field coverage. And if you're pulling data at scale, picking the wrong filter matters a lot.

I wanted to test this systematically: run the same logical query through different filter combinations, compare what came back, and figure out which filter gives you the most useful result set for our use case.

The problem was the API had a lot of filters. Writing a script for each permutation by hand would take hours. So I handed the docs to Claude and asked it to do the scaffolding.

The goal wasn't to find bugs. It was to understand how the API actually behaves — which filters produce clean data, which ones bloat the response, and which one is the right default for our integration.

02 · Step one

Export the docs, let Claude organise them

I opened the vendor's documentation URL, exported it to PDF, and dropped it into Claude. The prompt was simple:

Prompt to Claude natural language
"Here's the API documentation as a PDF. Parse it and create separate markdown files."

That's it. No instructions on how to structure them, no list of what to include. Claude figured out the right separation on its own — auth got its own file, each logical API layer got its own file, and the structure was consistent across all of them.

What came back was better than I expected. Claude read the entire PDF and produced a clean set of files:

api-docs/ ├── auth.md ├── users.md ├── data-endpoints.md ├── reporting.md └── webhooks.md

Each file had a consistent structure: the endpoint, the HTTP method, required auth headers, the request body shape, and what a success response looks like. auth.md specifically captured the token flow — what to call first, how the token is scoped, what the expiry behaviour is.

This step alone saved significant time. Instead of keeping the full PDF open and mentally parsing it while writing tests, I now had a clean, scannable reference that Claude itself could read back later.

03 · Step two

Turn the markdown files into a filter comparison script

With the structured markdown files in place, the next prompt was about what I actually wanted to learn — not just "run these endpoints" but "compare what different filters return":

Prompt to Claude Code natural language
"Read the markdown files in api-docs/. Create a Python script to match my hypothesis — use this filter and compare, and tell me which is better and what is the difference."

Again, no itemised instructions. Because the markdown files already captured exactly what filters existed and what each one did, Claude had enough context to understand what "compare" meant — same query, different filters, tell me what changes. It produced a script that ran each filter variant and captured the result shape side by side:

VOLUME

Record count per filter

Some filters returned 10x more records than others for an identical logical query. That's a cost and performance decision, not just a data one.

COVERAGE

Field population rate

Which fields were actually filled in vs consistently null across different filter paths — surfacing which filter gives you the richest dataset.

OVERLAP

Result set overlap between filters

Are two filters returning the same records by a different path, or genuinely different subsets? The answer changed how we thought about combining them.

PERF

Response time variance

One filter was documented the same as the others but consistently ran 3–4x slower. That's the kind of thing you don't see in docs — only in a comparison run.

04 · The output

A report on what each filter actually gives you

Claude wrote the Python scripts, ran them, and came back with a final report. Not pass/fail — a straight breakdown of what each filter returns and how the datasets differ.

Recommendation

Use by_entity_id when data completeness matters. Use by_category only if total volume is the priority — expect sparse fields.

The hypothesis was right: the filters weren't interchangeable. by_category looks powerful on paper — it returns the most records — but most of those records have large chunks of data missing. by_entity_id returns a smaller, cleaner dataset where almost every field is populated.

That's now a decision backed by actual data. Not a guess, not re-reading the docs — a script that ran the comparison and spelled out the difference.

The script is reusable. When the vendor changes something or adds a new filter, re-run it and the comparison updates automatically.

05 · Why it works

What makes this workflow click

The key move was the intermediate step — structured markdown before any script. Most people would go straight from "I have API docs" to "write me a comparison script." That shortcut usually produces shallow output — the model is trying to parse an unstructured document and generate code at the same time.

By first asking Claude to re-structure the docs into clean, per-domain files, you give it a better substrate to work from. The script generation step becomes almost mechanical — each markdown file is a spec, and the comparison logic follows directly from the filter parameters documented in it.

1

Raw docs → structured markdown

Claude reads the PDF and re-formats into consistent, per-domain files. Auth gets its own file. Each API group gets its own file. No mixing.

2

Markdown → test scenarios

With structured input, Claude can enumerate scenarios systematically — not just the happy path, but auth failures, validation errors, and edge cases per endpoint.

3

Tests → run → report

Claude Code runs the tests and surfaces the failures. The interesting ones are always the discrepancies between docs and actual behaviour.

06 · Tips

Things worth knowing if you try this

✅

Don't just feed the URL — prep the input first

Before pointing a coding agent at raw documentation, export it into a format it can actually work with. A URL is a starting point. A well-structured file is a working input.

✅

Coding agents read markdown files really well

Markdown is one of the most powerful inputs you can give a coding agent. Clean headings, consistent structure, no noise — it reads and reasons over it far better than a PDF or a web page.

✅

Structured output over raw dumps

When you ask Claude to parse docs, ask for structure — one file per domain, consistent sections. The quality of what the agent produces later is directly tied to how well the input was organised.

✅

Your prompt sets the goal, not the steps

You don't need to spell out every instruction. Give the agent a clear goal and trust it to figure out the how — especially when the context is already in the markdown files.
